# 漫画下载 SOP 复盘与优化方案

> 基于 `秘密教學` 大型连载（API 预期 9,485 张；旧流程仅统计到 8,934 张 / 322 MB）下载实战复盘。本文记录故障事实和最终实施方案；早期的 5% 容差、固定 3-5 册及 50 MB 分卷提议均已废弃。

## 实施状态（2026-06-21）

| 能力 | 当前实现 |
|---|---|
| 上游接入 | 自定义 `ValidatedMobilePdfFeature` 注入 `download_album(..., extra=feature)`，复用原 album/downloader 生命周期 |
| 图片发现 | 按 API 章节顺序和 `page_arr` 文件名递归扫描规范目录，并排除其他章节子目录 |
| 完整性 | 每章 `valid == expected` 且空/损坏文件为 0；不接受百分比容差 |
| 状态 | 不完整返回 `partial` 且 `output_paths=[]`；Feature 未执行也不能沿用上游 `success` |
| PDF | 图片逐张压缩、`img2pdf` 流式写入、`pypdf` 精确验页，按章节边界规划分卷 |
| 飞书 | 每卷 18 MB 安全线，只发送校验通过的 PDF，任何 ZIP 请求强制进入 PDF 恢复管线 |
| 续传 | 最多 5 轮，15/30/60 秒退避；连续两轮有效图片数和完成章节数均不增长即停止 |
| 状态存储 | `/root/.hermes/state/jmcomic/<album_id>/manifest.json` |
| 目录规则 | 强制 `Bd / JM{Aid}-{Pid}`，同名章节不再覆盖，标题变化也不改变目录 |

## 一、现有 SOP 现状速览

| 文件 | 位置 | 当前覆盖范围 |
|---|---|---|
| `hermes/SOUL.md` | 第 157-180 行 | 检索 → 下载 → `post_process(img2pdf)` → `MEDIA:` 回传飞书 |
| `docs/integration-handoff.md` | 第 1-318 行 | 集成部署、配置注入、环境验证、PDF-only wrapper 说明 |

**现有 SOP 核心假设**（与实际不符）：
- 假设 `post_process(img2pdf, level="album")` 能正确处理全部图片 → **实际只扫主目录，漏 297 个章节子目录**
- 假设 `download_album` 一次完成 → **实际大型连载需 3 轮重试**
- 假设图片格式为 `.jpg/.png` → **实际为 `.webp`**
- 假设 `post_process` 成功则无需 fallback → **实际手动 PIL 分批 500 张才勉强跑完**
- 假设飞书分卷自动生成合理 → **实际 18 个 20MB 分卷，用户体验差**

## 二、6 大问题复盘与根因分析

### 1. 🔴 post_process 严重缺陷：只扫主目录，漏掉章节子目录

| 指标 | 数据 |
|---|---|
| 旧磁盘统计 | 8,934 张 |
| API 精确预期 | 9,485 张 |
| post_process 处理 | 125 张（仅主目录） |
| 漏掉 | 8,809 张（297 个章节子目录） |
| 处理率 | **1.4%** |
| 返回状态 | `success`（无警告） |

**根因**：`post_process` 的 `level="album"` 只扫描专辑根目录，**不递归子目录**。章节子目录里的图片被完全忽略。且没有校验机制，导致返回 `success` 却几乎什么都没做。

---

### 2. 🔴 下载超时无重试策略：大型连载 600s 不够，跑了 3 轮

| 轮次 | 进度 | 结果 |
|---|---|---|
| 第 1 轮 | 下到 ~122 话 | MCP 600s 超时 |
| 第 2 轮 | 下到 ~233 话 | 被用户消息中断 |
| 第 3 轮 | 完成剩余 | 成功 |

**根因**：SOP 只说"下载是阻塞操作，会实时回报进度"，但**没定义超时后的行为**：
- 重试几次？
- 每次间隔多久？
- 怎么判断已下载 vs 未下载？
- 怎么续传而不是从头下？

---

### 3. 🟡 图片格式踩坑：find `*.jpg` / `*.png` 返回 0，实际是 `.webp`

**根因**：SOP 没有提及图片格式可能是 `.webp`，也没有给出**通用图片统计命令**（如 `find -type f | grep -iE '\.(jpg|jpeg|png|webp|gif|bmp)$'`）。

---

### 4. 🟡 手动转 PDF 反复超时：8934 张一次性失败，500 张一批才勉强

| 尝试 | 规模 | 超时 | 结果 |
|---|---|---|---|
| 一次性 | 8,934 张 | 300s | 超时 |
| 分批 | 3,000 张/批 | 300s | 超时 |
| 分批 | 500 张/批 | 600s | 勉强完成 |

**根因**：SOP 完全没有 `post_process` 失败时的 **fallback PDF 生成方案**。用户被迫现场试错，靠缩减批次才找到可行阈值。

---

### 5. 🟡 章节目录命名不规范：难以批量处理

下载后的目录名格式不统一：

| 示例 | 格式 | 问题 |
|---|---|---|
| `秘密教學 10` | 空格+话数 | 空格分隔，话数无前缀 |
| `秘密教學_第100話_-_爭風吃醋的3P遊戲` | 下划线+话数+标题 | 含特殊符号、日文假名 |
| `秘密教學 147 給乖巧奴隸的獎賞` | 空格+话数+标题 | 无统一分隔符 |
| `秘密教學114` | 无分隔符 | 话数直接拼接 |
| `秘密教學 連載調整告知` | 非章节内容 | 无数字，无法识别为章节 |

**根因**：服务器原配置为 `dir_rule.rule: Bd_Pname`。目录名不仅不规范，更严重的是 20 个同名普通章、22 个“休刊公告”和 4 个“停刊公告”映射到共享目录，后下载章节覆盖先下载章节。最终规则改为 `Bd / JM{Aid}-{Pid}`；章节顺序取 API，不再解析目录名。

---

### 6. 🟡 飞书分卷过多：18 个 20MB 文件，逐个发送体验差

| 指标 | 数据 |
|---|---|
| 总量 | 322 MB |
| 分卷数 | 18 个 |
| 单个大小 | ~20 MB（接近飞书实际限制） |
| 发送方式 | 逐个 `MEDIA:` 上传 |

**根因**：wrapper 按 28MB 自动分卷，但未考虑**按章节逻辑分卷**（如 1-10 话为一册、11-20 话为一册），导致用户收到 18 个无意义编号的 part 文件，无法对应阅读进度。

---

## 三、优化方案：SOP v2.0

### 总体思路

1. **下载阶段**：改为**章节级续传**，而非全专辑重试；支持超时后自动断点续传。
2. **后处理阶段**：放弃 `post_process` 的 `level="album"`，改用**递归扫描 + 章节级 PDF**（每章一个 PDF，或按章节范围合并为多册）。
3. **校验阶段**：下载后、转 PDF 前、回传前，**三处强制计数校验**。
4. **回传阶段**：保持章节边界并以 18 MB 为硬安全线动态分卷；分卷数由实际压缩体积决定。

---

### 3.1 优化一：下载超时重试策略（解决 #2）

**新增 SOP 步骤**：

```markdown
### 下载超时与续传策略

1. **预评估**：`get_album_detail` 返回章节数 > 20 时，向用户预告"本专辑共 N 话，预计需多轮下载，每轮约 10-15 话"。
2. **超时判断**：`download_album` 若返回 timeout / 进度未达 100%，**不要立即重新调用全专辑下载**——这会从头下，浪费带宽。
3. **断点续传**：
   - 检查已下载目录：`ls /tmp/monitor_charts/jmcomic/<album_dir>/`
   - 已完成的章节子目录存在且有图片 → 跳过
   - 未完成的章节 → 用 `download_photo(photo_id=...)` 单章补下
   - 或者直接用 `download_album` 的**幂等性**（jmcomic 库默认会跳过已存在文件），直接重试即可，但需在调用前向用户确认
4. **重试上限**：最多 5 轮，退避 15/30/60 秒；连续两轮无进展提前停止。达到上限仍不完整时只报告缺失，不发送部分 PDF。
5. **进度汇报**：每轮结束后汇报"已完成 X/Y 话，共 Z 张图"。
```

---

### 3.2 优化二：post_process 替代方案——递归扫描 + 章节级 PDF（解决 #1、#4、#5）

**核心改动**：不再信任 `post_process(img2pdf, level="album")`，改为**自定义 Python 脚本**接管后处理。

**新增 SOP 步骤**：

```markdown
### 后处理：递归扫描所有图片，按章节合并为 PDF 分册

1. **枚举所有章节目录**：
   ```bash
   # 通用图片统计（不依赖格式）
   find /tmp/monitor_charts/jmcomic/<album_dir>/ -type f \
     | grep -iE '\.(jpg|jpeg|png|webp|gif|bmp)$' | wc -l
   ```

2. **校验完整性**：
   - 统计预期图片数（从 `get_album_detail` 的章节页数之和计算）
   - 每章实际有效图片数必须精确等于 API 预期数；少 1 页也返回 `partial`，先补下载再转 PDF

3. **PDF 生成策略（Python 脚本，替代 post_process）**：
   - 遍历每个**章节子目录**（跳过无图片的目录）
   - 每章单独转为一个临时 PDF（避免 8000+ 张大图一次性加载）
   - 按 API 章节顺序累加，在不超过 18 MB 安全线的前提下尽量保持章节完整
   - 输出命名：`{标题}_第001-050話.pdf`、`{标题}_第051-100話.pdf`...

4. **恢复入口**：再次调用 `post_process(album_id, "img2pdf")`，它会复用同一清单和 PDF 管线。禁止使用未经页数校验的 PIL 或 shell 通配符临时产物作为交付附件。
```

---

### 3.3 优化三：章节目录规范化处理（解决 #5）

目录名只用于定位文件，不再承担章节解析职责：

```markdown
### 章节目录处理

- 不重命名下载目录，也不使用正则猜测章节号。
- 章节 ID、标题、顺序和预期页数全部来自 jmcomic album/photo 对象。
- 非数字章节（如 `連載調整告知`）仍按 API 顺序保留；仅当 API 本身不包含该目录时才视为孤立文件。
```

---

### 3.4 优化四：飞书回传分册策略（解决 #6）

**核心改动**：使用章节范围命名，但仍以飞书实测可用的 **18 MB 安全线**决定动态分卷数。

**SOP 修改**：

```markdown
### 飞书回传：按章节范围分册，而非按文件大小分卷

1. **分册命名规则**：`{标题}_第001-050話.pdf`、`{标题}_第051-100話.pdf`...
2. **分册规则**：按 API 顺序累计完整章节，预估达到 90% 安全容量时换卷；生成后再次检查实际文件不超过 18 MB。
3. **MEDIA: 输出**：每个分册一行 `MEDIA:/tmp/monitor_charts/jmcomic/标题_第xxx-xxx話.pdf`。
4. **用户提示**：发送时附带说明"本专辑共 N 册，第 1-50 话 → 第 51-100 话..."，方便用户按顺序阅读。
```

---

### 3.5 优化五：三处校验检查点（解决 #1、#3）

**SOP 新增校验步骤**：

```markdown
### 三处校验检查点（强制）

| 检查点 | 时机 | 方法 | 失败处理 |
|---|---|---|---|
| **检查点 1：下载后** | `download_album` 返回后 | 按 album/photo 元数据逐章递归扫描并验证图片 | 任一章不精确匹配 → `partial`，重试补下 |
| **检查点 2：PDF 生成后** | 每个分册生成后 | 用 PyPDF2 / pdfinfo 读取 PDF 页数，对比该分册包含的图片数 | 页数 ≠ 图片数 → 重新生成该分册 |
| **检查点 3：回传前** | `MEDIA:` 前 | `ls -lh` 确认文件存在且大小 > 0 | 文件缺失/空 → 重新生成 |
```

## 四、SOP v2.0 完整流程图

```
用户请求
  ↓
search_album / get_album_detail → 确认专辑 ID、章节数、预期总页数
  ↓
【预评估】章节数 > 20 → 向用户预告"预计多轮下载"
  ↓
download_album（600s timeout）
  ↓
超时？ → 检查已下载目录 → 续传未完成章节（download_photo 或重试 download_album）
  ↓
【检查点 1】持久化清单逐章精确比较有效图片数与预期页数 → 任一差异即补下
  ↓
使用 album/photo 元数据建立章节顺序与目录映射
  ↓
按组生成 PDF（每章先转临时 PDF，再合并为分册）
  ↓
【检查点 2】pdfinfo 页数 vs 图片数 → 不符则重生成
  ↓
【检查点 3】ls -lh 确认文件存在
  ↓
MEDIA: 回传飞书（附带分册说明）
  ↓
汇报：标题、ID、总话数、总图片数、分册列表、各分册话数范围
```
